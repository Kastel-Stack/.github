# Kastel Stack

**Human-supervised adaptive business infrastructure for sovereign, digitally legible enterprises.**

Start here:

1. `kastel-ground-truth` - canonical design doctrine
2. `kastel-stack` - public framework and reference implementation
3. `kastel-node-template` - starter for private node deployments

Kastel Stack is an open, auditable framework for helping independent operators, small teams and cross-border ventures model their business, connect their tools, detect bottlenecks, propose safe actions, preserve human judgement, and bank what works as reusable operating capability.

It is designed for founder-led businesses, specialist teams, creator enterprises, research-led ventures, artisan businesses and decentralised operator networks that want intelligent growth without surrendering agency to black-box platforms, hyperscaler lock-in, venture-capital growth logic or unsupervised agentic automation.

Kastel Stack is not an automation stack in the narrow sense.

It is a **supervised Gf → Gc conversion system for business operations**.

---

## Core idea

Kastel treats every business as a **scaling model hypothesis**.

A business is not simply optimised. It is tested, clarified and improved through supervised adaptive loops:

```text
offer
+ niche
+ proof
+ route to market
+ delivery loop
+ feedback system
+ governance capacity
```

The first question is not:

```text
How do we optimise this business?
```

It is:

```text
What scaling model are we testing?
```

The goal is not to automate everything.

The goal is to help a business:

```text
model itself clearly
detect real bottlenecks and opportunities
diagnose state–niche fit
propose gated actions or clarification moves
observe source-of-truth outcomes
test whether patterns transfer
bank only what survives
```

Kastel is therefore a **Gf (fluid intelligence) → Gc (crystallised intelligence) conversion engine for business operations**:

```text
adaptive problem-solving under uncertainty
→ tested routines
→ reusable scripts
→ operating schemas
→ strategic invariants
```

A win is not banked simply because it worked once.

Kastel banks only patterns that survive evidence, review, boundary conditions, niche-context checks and, where feasible, wrapper, segment, channel or timing variation.

## Ground-truth map

The Kastel operating map is:

```text
external tools and business systems
→ connector layer
→ typed events and metric snapshots
→ measurement and semantic spine
→ business scaling model hypothesis
→ variable abstraction
→ business state map
→ external niche map
→ state–niche fit estimate
→ G Kernel diagnosis
→ Action Intent / Clarify Intent
→ Auto / Draft / Escalate / Blocked gate
→ human cockpit or safe automation
→ source-of-truth outcome observation
→ wrapper / segment / channel / niche-context test
→ delayed review
→ script banking or retirement
→ slow invariant extraction
```

Or more compactly:

```text
Signals become events.
Events become variables.
Variables become maps.
Maps become diagnoses.
Diagnoses become Action or Clarify Intents.
Intents pass through gates.
Outcomes are observed.
Portable patterns are banked.
Cross-run invariants reshape future judgement.
```

The kernel does not interface with raw data streams in a loose sense.

It interfaces with:

```text
typed events
metric snapshots
semantic objects
source-of-truth records
connector-normalised signals
human decision records
outcome observations
```

This prevents the system from confusing noise, trend language or tool outputs with business truth.

## What we are building

Kastel Stack is being developed as a practical, open framework for a supervised adaptive scaling operating system.

Core components include:

- **G Kernel** — a Trident-G-aligned inference and routing layer
- **State–Niche Fit** — diagnosis of internal readiness and external legibility
- **Action Intents** — proposed interventions routed through human gates
- **Clarify Intents** — information-gathering moves before premature action
- **Measurement & Semantic Spine** — source-of-truth events, definitions and data lineage
- **External Niche Map** — audience, partner, platform, market, regulatory and cultural context
- **Policy Pack** — rules for Auto, Draft, Escalate and Blocked actions
- **Script Library** — validated operating patterns and reusable Φ-Scripts
- **Connector Pack** — typed integrations with tools such as GitHub, Supabase, Stripe, Brevo, Cloudflare and others
- **Human Cockpit** — review interfaces for judgement-sensitive decisions, approvals, workload and governance

The G Kernel receives typed, source-of-truth-grounded signals from business systems, maps them onto a semantic model of the venture, diagnoses internal state and external niche fit, and generates prioritised Action or Clarify Intents.

Low-risk, reversible and evidence-backed actions may be automated.

Judgement-sensitive actions are routed to human review.

## Action and clarification logic

Every proposed move should become either an **Action Intent** or a **Clarify Intent**.

An **Action Intent** is a proposed intervention.

Examples include:

```text
publish a revised landing page
send a CRM rescue email
fix an onboarding friction point
launch a small proof-led content test
invite a partner to a pilot call
update a claim-sensitive proof page
```

A **Clarify Intent** is an information-gathering move used when the external niche is too opaque for direct action.

Examples include:

```text
ask a customer what confused them
separate two audience segments in analytics
request clearer partner terms
verify whether a search trend reflects buyer intent or curiosity
review claim boundaries before launching a page
compare two channels before committing effort
check whether a partner route creates trust or capture risk
```

The rule is:

```text
If the action would not teach us anything interpretable,
clarify before acting.
```

---

## Human-gated autonomy

Kastel uses bounded autonomy.

Each Action Intent is routed through one of four gates:

| Gate | Meaning |
|---|---|
| **Auto** | Low-risk, reversible, routine and evidence-backed action |
| **Draft** | Useful action prepared for human review before external use |
| **Escalate** | Judgement-sensitive action involving claims, money, privacy, partnerships, strategy, security, compliance or reputation |
| **Blocked** | Action not permitted under current policy |

The system may assist with interpretation, modelling, drafting, classification and routing.

It does not own business truth or bypass human judgement.

Critical actions remain supervised.

## Source-of-truth discipline

Kastel depends on a strict source-of-truth rule.

Each important business object should have one primary owner system.

Examples:

```text
GitHub = code and specification truth
Supabase = operational truth
Stripe = payment truth
Brevo = CRM and email truth
Cloudflare = web infrastructure truth
Google Search Console = organic discovery truth
QuickFile / Xero / QuickBooks = accounting truth
Gmail / helpdesk = correspondence and support truth
```

Other systems may mirror, cache, enrich, report or trigger workflows from these facts.

They should not silently become the source of truth.

The rule is:

```text
No semantic clarity, no automation.
No source-of-truth path, no banking.
No lineage, no model-owned recommendation.
No niche context, no strategic invariant claim.
```

## State–niche fit

State–niche fit is the core Kastel diagnostic.

A business is in state–niche fit when:

```text
the business is internally ready enough to act
+
the niche is externally legible enough to generate useful feedback
+
the action can be observed through a source-of-truth outcome
+
the human operator has enough judgement capacity to supervise the loop
```

The kernel should avoid direct action when any of these are weak.

The practical question is:

```text
Is the business ready to act,
and is the external niche legible enough
for the action to generate useful learning?
```

If the answer is yes, the system may propose an Action Intent.

If the answer is no, the system should propose a Clarify Intent, a reset, a simplification, a boundary check or a pause.

## What Kastel is not

Kastel Stack is not:

- a generic chatbot
- an unsupervised agent swarm
- a black-box growth optimiser
- a replacement for business judgement
- a legal, financial, clinical or regulatory decision engine
- an enterprise surveillance stack
- a system for pursuing runaway growth at any cost
- a system for banking one-off wins as permanent truth
- a top-down management system disguised as AI governance

Kastel is designed to make human judgement easier, not bypass it.

---

## Philosophy

Kastel Stack is based on **polycentric platform sovereignty**.

This means using digital platforms, cloud services, AI systems and markets pragmatically, while avoiding unnecessary dependence on any single platform, vendor, funder, jurisdiction or institutional logic.

The aim is not isolation.

The aim is increasing operational sovereignty:

```text
sovereign enough to refuse capture
connected enough to learn
adaptive enough to grow
supervised enough to preserve judgement
```

Kastel is not anti-platform, anti-state or anti-technology.

It is anti-capture.

## Current status

Kastel Stack is in early specification and prototype-planning stage.

The first implementation target is a lightweight stack using:

```text
GitHub for code and specification truth
Supabase for operational truth
Docker on a local or small-server runtime node
typed connector modules
human review interfaces
state–niche diagnostics
Action / Clarify Intent routing
Auto / Draft / Escalate / Blocked gates
script banking and delayed review
```

The first practical deployment is being developed through IQ Mindware / HRP Lab as a live test case for evidence-led, human-supervised business scaling.

The first operational target is one reliable closed loop:

```text
define the business scaling model hypothesis
→ identify the proposed strategic invariant
→ instrument the business
→ define key events and semantic objects
→ detect one bottleneck, opportunity or uncertainty source
→ diagnose internal state and external niche legibility
→ decide whether to act or clarify
→ route through a safety gate
→ execute, draft, escalate or clarify
→ observe source-of-truth outcomes
→ test portability where feasible
→ review after delay
→ bank, revise or retire the script
```

## Security and privacy

Never commit:

```text
API keys
customer data
private emails
payment exports
personal information
production secrets
private client strategy notes
```

Security, privacy and compliance are treated as trust infrastructure, not afterthoughts.

The system should be designed so that public specifications, protocols and architecture can be open while private deployments, customer data, operational secrets and sensitive business logic remain protected.
